# Service Token Revocation System

## Overview

The Service Token Revocation System provides a secure mechanism to revoke service tokens used for service-to-service authentication. This addresses CRITICAL #2 from the security audit: "Service token has no revocation mechanism."

## Problem Statement

Previously, service tokens:
- Could not be revoked once issued
- Had no audit logging (explicitly disabled to avoid spam)
- Used a single static environment variable
- Were vulnerable to indefinite use if compromised

## Solution Architecture

### Components

1. **Revocation Storage** (`APIAuthService._revoked_service_tokens`)
   - In-memory set of SHA-256 token hashes
   - Fast O(1) lookup performance
   - Suitable for single-instance deployments
   - **Note**: For multi-instance deployments, consider Redis or database storage

2. **Revocation Endpoint** (`POST /v1/auth/service-token/revoke`)
   - Requires SYSTEM_ADMIN role
   - Accepts token and reason
   - Returns success status and partial hash

3. **Validation Check** (`APIAuthService.validate_service_token`)
   - Checks revocation BEFORE token comparison
   - Prevents timing attacks
   - Logs failed attempts with IP and user agent

4. **Audit Logging**
   - All failed authentication attempts logged
   - Revocation events logged with reason and actor
   - Partial hashes used for security

## Implementation Details

### Token Hashing

```python
def _hash_service_token(self, token: str) -> str:
    """Hash a service token for revocation storage.

    Uses SHA-256 for fast lookups. This is safe because:
    1. Service tokens are high-entropy (generated by manager)
    2. We use constant-time comparison for validation
    3. Hash is only used for revocation checks, not authentication
    """
    return hashlib.sha256(token.encode("utf-8")).hexdigest()
```

### Revocation Check

```python
def is_service_token_revoked(self, token: str) -> bool:
    """Check if a service token has been revoked."""
    token_hash = self._hash_service_token(token)
    return token_hash in self._revoked_service_tokens
```

### Revocation Method

```python
async def revoke_service_token(self, token: str, reason: str, revoked_by: str) -> bool:
    """Revoke a service token.

    Returns:
        True if revocation succeeded, False if token was already revoked
    """
    token_hash = self._hash_service_token(token)

    if token_hash in self._revoked_service_tokens:
        return False

    self._revoked_service_tokens.add(token_hash)

    # Audit logging
    logger.info(f"[AUTH] Service token revoked: hash={token_hash[:8]}..., reason={reason}, revoked_by={revoked_by}")

    return True
```

### Validation Flow

```python
def validate_service_token(self, token: str) -> Optional[User]:
    """Validate a service token and return a service account user."""

    # Check revocation FIRST
    if self.is_service_token_revoked(token):
        logger.warning("[AUTH] Service token validation failed: token is revoked")
        return None

    # Standard validation
    expected_token = os.environ.get("CIRIS_SERVICE_TOKEN")
    if not expected_token or not hmac.compare_digest(token, expected_token):
        return None

    # Return service account user
    return User(...)
```

## API Usage

### Revoke a Service Token

**Endpoint**: `POST /v1/auth/service-token/revoke`

**Authentication**: Requires SYSTEM_ADMIN role

**Request**:
```json
{
  "token": "service_token_to_revoke",
  "reason": "Security incident - token potentially compromised"
}
```

**Response**:
```json
{
  "success": true,
  "message": "Service token revoked successfully",
  "token_hash": "a1b2c3d4e5f6789..."
}
```

**Example**:
```bash
curl -X POST http://localhost:8080/v1/auth/service-token/revoke \
  -H "Authorization: Bearer YOUR_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "token": "service_token_to_revoke",
    "reason": "Token rotation - replacing with new token"
  }'
```

## Security Features

### 1. Audit Logging

**Failed Authentication Attempts**:
```python
# From auth.py dependencies
logger.warning(
    f"[AUTH SECURITY] Failed service token authentication: "
    f"token_hash={token_hash}, client_ip={client_ip}, "
    f"user_agent={request.headers.get('user-agent', 'unknown')}"
)
```

**Revocation Events**:
```python
logger.info(
    f"[AUTH] Service token revoked: hash={token_hash[:8]}..., "
    f"reason={reason}, revoked_by={revoked_by}"
)
```

### 2. Timing Attack Protection

- Uses `hmac.compare_digest()` for constant-time comparison
- Revocation check happens BEFORE comparison
- No information leakage about token validity

### 3. Hash Privacy

- Only partial hashes (8-16 characters) logged
- Full hashes never exposed in responses
- SHA-256 ensures one-way transformation

### 4. Role-Based Access Control

- Only SYSTEM_ADMIN users can revoke tokens
- Authorization check enforced at endpoint level
- Prevents unauthorized revocation

## Deployment Considerations

### Single-Instance Deployments

The in-memory revocation set works perfectly for single-instance deployments:
- Fast O(1) lookups
- No external dependencies
- Automatic cleanup on restart

### Multi-Instance Deployments

For horizontal scaling with multiple API instances, consider:

1. **Shared Cache (Redis)**:
   ```python
   async def revoke_service_token(self, token: str, reason: str, revoked_by: str) -> bool:
       token_hash = self._hash_service_token(token)
       await redis_client.sadd("revoked_service_tokens", token_hash)
       return True

   def is_service_token_revoked(self, token: str) -> bool:
       token_hash = self._hash_service_token(token)
       return await redis_client.sismember("revoked_service_tokens", token_hash)
   ```

2. **Database Table**:
   ```sql
   CREATE TABLE revoked_service_tokens (
       token_hash TEXT PRIMARY KEY,
       revoked_at TIMESTAMP,
       revoked_by TEXT,
       reason TEXT
   );
   ```

3. **Message Bus**:
   - Broadcast revocation events
   - Each instance updates local cache
   - Eventually consistent

## Best Practices

### Token Rotation

Rotate service tokens regularly:

```bash
# 1. Generate new token
NEW_TOKEN=$(openssl rand -base64 32)

# 2. Update manager with new token
# (Manager-specific procedure)

# 3. Wait for propagation

# 4. Revoke old token
curl -X POST .../revoke -d '{"token":"OLD_TOKEN","reason":"Scheduled rotation"}'
```

### Incident Response

If a service token is compromised:

1. **Immediate Revocation**:
   ```bash
   curl -X POST .../revoke -d '{"token":"COMPROMISED_TOKEN","reason":"Security incident"}'
   ```

2. **Check Audit Logs**:
   ```bash
   grep "Failed service token authentication" /app/logs/incidents_latest.log
   ```

3. **Rotate Token**:
   - Generate new token
   - Update all services
   - Monitor for unauthorized attempts

4. **Review Access Patterns**:
   - Check which IPs attempted to use the token
   - Investigate suspicious user agents
   - Look for anomalous access patterns

### Monitoring

Monitor for suspicious activity:

```bash
# Failed authentication attempts
grep "AUTH SECURITY.*Failed service token" /app/logs/incidents_latest.log

# Recent revocations
grep "Service token revoked" /app/logs/incidents_latest.log

# Count failed attempts by IP
grep "Failed service token" /app/logs/incidents_latest.log | \
  awk -F'client_ip=' '{print $2}' | awk '{print $1}' | sort | uniq -c
```
